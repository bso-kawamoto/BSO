# AGENTS.md

## プロジェクト目的

BSO Operation は、イベントが立て込む社内業務を、案件、タスク、担当者、予定の単位で整理するための業務管理MVPです。最初の目的は、案件ごとの進捗と期限を一画面で把握できる状態を作ることです。

## 開発原則

- シンプル優先
- 過剰設計禁止
- 既存構造を壊さない
- 変更は目的に直結する範囲に絞る
- `npm run build` が通ること
- TypeScript型安全を重視する

## 使用技術固定

- Next.js App Router
- React
- TypeScript
- Supabase
- Vercel
- npm

## UI原則

- 実務で毎日見る画面として、読みやすさと操作しやすさを優先する
- 装飾よりも一覧性、状態の見やすさ、入力のしやすさを優先する
- カードは情報単位に使い、画面全体を過度に飾らない
- モバイルでもフォームとボタンがはみ出さないようにする

## 情報設計ルール

- 案件の中に中タスクを置き、小タスクは中タスクの配下に置く
- 担当者は社員マスターから選択する
- 予定はタスクとは別に `calendar_events` で管理する
- ボード内カレンダーでは予定名を表示し、案件・タスクの期日は件数で表示する
- 通常ボードではアーカイブ済み案件を非表示にする

## MVP範囲

- 案件登録、案件名・期日編集、案件アーカイブ
- 中タスク・小タスク登録
- タスクの担当者、ステータス、カテゴリ、優先度、期日管理
- タスクの進捗メモ管理
- 案件詳細でのタスク・予定の追加、編集、削除
- カレンダー予定登録、複数日、終日、時間指定
- カレンダーの月移動、予定詳細、予定編集、予定削除
- 社員別ビュー
- 今日やることビュー
- 社長・河本向け管理画面
- Supabase未設定時のサンプル表示
- Supabase Authによるメールアドレス・パスワードログイン

## コード規約

- TypeScript strict を維持する
- 型は `lib/types.ts` と `lib/supabase/types.ts` を更新する
- DB固定値とUI固定値をずらさない
- 不要な抽象化を増やさない
- コメントは複雑な処理の補助に限定する

## DBルール

- Supabase前提
- スキーマ変更は `supabase/migrations/` にSQLとして追加する
- `SUPABASE_SERVICE_ROLE_KEY` はServer Actions専用で扱い、クライアントに出さない
- 社員ログインは `employees.email` または `employees.auth_user_id` でSupabase Authユーザーと紐づける
- 社長など社員マスター外の管理者は `APP_ADMIN_EMAILS` で管理する
- RLSは有効化し、MVPでは必要最小限の読み書きを許可する
- `projects.is_archived` で案件の通常表示/非表示を管理する
- `operation_tasks.memo` は進捗メモとして使い、長文履歴や監査ログとは分ける

- `未着手`
- `進行中`
- `確認待ち`
- `完了`

## カテゴリ固定

- `営業`
- `運用`
- `請求`
- `サポート`
- `管理`

## UXルール

- エラー時も画面表示を維持する
- フォームには必ずラベルを付ける
- 登録、更新、削除後は分かる場所に通知を出す
- 登録済み予定は詳細画面で確認・編集・削除できるようにする
- 期日が近いもの、期限切れのものを見つけやすくする
- ログイン後、社長と河本は全体表示、その他社員は本人分だけ表示する

## Codex変更ルール

- 既存構造を壊さない
- 無関係なリファクタをしない
- 変更後は `npm run lint` と `npm run build` を確認する
- READMEに影響がある変更はREADMEも更新する
- 生成・変更したファイルを最後に報告する

## デプロイルール

- Vercel前提
- デプロイ前に `npm run build` を通す
- Vercel環境変数にSupabase URL、Anon Key、必要に応じてService Role Keyを設定する
- 本番Supabaseにマイグレーションを適用してから公開する

## README更新ルール

- セットアップ、環境変数、DBスキーマ、ディレクトリ構成に変更がある場合はREADMEを更新する
- 新しい主要画面や主要機能を追加した場合はREADMEに追記する
